## 3. Cross-Cutting Issues

These problems appear on 3 or more screens — they indicate template / shared component bugs.

### CC-01 · FormField label not linked to input (affects ALL forms — 12+ screens)

**Root cause:** `FormField` (`shared/form/FormField.tsx:36`) renders `<Label>{resolvedLabel}</Label>` but no `id` is set on the child `<Input>` and no `htmlFor` is passed to `<Label>`. The `ShadLabel` component renders a `<label>` element, but without `htmlFor` → `id` wiring, clicking the label does not focus the input and screen readers cannot programmatically associate them.

**Affected screens:** All form screens — `/events/new`, `/events/[id]` (edit), `/sports/new`, `/sports/[id]`, `/organizations/new`, `/organizations/[id]`, `/users/new`, `/users/[id]`, `/register/new` (all steps), `/participation/new`, `/surveys/[id]/*`, survey count inputs in BySportSurveyForm.

**WCAG impact:** Fails 1.3.1 (Info and Relationships) and 1.1.1 (Non-text Content) — level A violation.

### CC-02 · Sidebar is fixed 224px with no mobile collapse (affects ALL portal screens — 29 screens)

**Root cause:** `Sidebar` (`modules/common/components/Sidebar.tsx:67`) is `w-56 flex-shrink-0`. `CommonLayout` does not implement a hamburger/drawer toggle. On 375px viewport, sidebar takes 150px of 375px, leaving 225px for page content — all tables become effectively unreadable.

**Affected screens:** All authenticated portal screens.

### CC-03 · DataTable pagination buttons lack `aria-label` (affects all paginated lists — 7 screens)

**Root cause:** `DataTable` (`shared/ui/DataTable.tsx`) renders prev/next as `<Button>›</Button>` and `<Button>‹</Button>`. The Unicode arrow characters are the only content — screen readers announce "button button" with no indication of direction or page.

**Affected screens:** `/events`, `/sports`, `/organizations`, `/users`, `/register`, `/participation`, `/submissions` (if paginated).

### CC-04 · No empty-state detail message in DataTable (affects all lists)

**Root cause:** When `DataTable` renders zero rows it shows a bare `—` (em dash) in a `<td>` (line 76–82). This only appears if `QueryBoundary` does not catch the empty case first. When `QueryBoundary.empty` is not passed, the DataTable dash is shown — not actionable or informative.

### CC-05 · No "last updated" / data freshness indicator (affects all data screens)

No screen communicates when data was last loaded or whether it is stale. React Query's `staleTime` defaults mean data could be up to 60 seconds old on dashboard with no visual cue.

## 4. Detailed Findings Index

### Information Completeness

**#B01 · EventList status column mislabeled as "Actions"**
- File: `modules/events/components/EventList.tsx:63`
- Column 5 header: `header: tc('actions')` — renders the `EventStatusBadge` but is labeled "Actions" (Actions). Column 6 renders the actual action buttons with header `''`.
- Fix prompt: Change column 5 header from `tc('actions')` to `t('columns.status')` (add translation key `events.columns.status`). Add a translated header to column 6 actions column, or explicitly mark it `header: ''` with `aria-label` on each action button (buttons already have `aria-label` ✓).

**#B02 · EventDetailPage: event type shown as raw Khmer enum, no label for "Type"**
- File: `modules/events/components/EventDetailPage.tsx:57–68`
- The `dl` list does not include `type` as a term/definition row. The `EventType` enum values are long Khmer strings displayed inline in `EventList` but not on the detail page at all.
- Also: no total participant count for the event, no registration window status indicator.

**#B03 · SurveyAdminTab: no sport_name column; no pagination**
- File: `modules/survey/components/SurveyAdminTab.tsx`
- Columns shown: org, athlete M/F, leader M/F, submitted_at. The `sport_id` or sport name is absent — admin cannot see which sport each survey row is for when multiple sports exist for an event.
- No pagination — fetches `limit: 200` silently; no indication if truncated.

**#B04 · UserForm: `is_active` toggle not rendered**
- File: `modules/users/components/UserForm.tsx:119–161`
- The form schema and `defaultValues` include `is_active`, and the submit handler passes it to the API. However, the JSX renders no `<input type="checkbox">` or toggle for this field. Admins cannot change a user's active/inactive status through the UI — it will always be the schema default (`true` for create, existing value for edit, but unchangeable).

**#B05 · RegistrationHomePage and OrganizerList: no event/sport filter**
- Files: `modules/registration-flow/components/RegistrationHomePage.tsx`, `modules/participation/components/OrganizerList.tsx`
- Both lists fetch all participants for the organisation (`role=athlete`/`role=leader`) without event or sport scoping. An org managing participants across multiple events sees everyone in one unfiltered table.

**#B06 · PersonalInfoStep: Khmer and English name fields share identical label keys**
- File: `modules/registration-flow/components/PersonalInfoStep.tsx:32–37`
- `kh_family_name` and `en_family_name` both have `labelKey="registration.fields.familyName"`. Both rows render as " នាមត្រកូល" (Family Name) with no visual indicator of which is Khmer and which is English. User can confuse which row to fill in Khmer script.

**#B07 · Placeholder pages: hardcoded English, no data shown**
- Files: `app/(portal)/register/[enroll_id]/page.tsx`, `app/(portal)/participation/[enroll_id]/page.tsx`
- Both show `"Participant #N"` / `"Organizer #N"` with an English hardcoded message. No i18n, no data fetch. Should be clearly marked as "Coming soon" via the standard `OrgIdGapBanner` or `UserWelcome` pattern, and use translated strings.

**#B08 · SubmissionList: FLAGGED and PENDING badges visually identical**
- File: `modules/submissions/components/ReviewActions.tsx:10–16`
- `STATUS_VARIANT` maps both `PENDING` and `FLAGGED` to `variant='outline'`. A FLAGGED submission (requiring attention) looks the same as a new PENDING one. This is a critical UI distinction for admin reviewers.

**#B09 · SubmissionList: no event filter, no pagination**
- File: `modules/submissions/components/SubmissionList.tsx:101`
- `DataTable` is rendered with no `pageCount`/`pagination` props — all up to 200 entries are shown in a single unscrolled table. No event/org filter is exposed on the list page.

**#B10 · SubmissionDetail: sport shown as raw `sport_id` number**
- File: `modules/submissions/components/SubmissionDetail.tsx` (via agent summary)
- The submission detail shows metadata including `sport_id` as a raw integer. No lookup to display the sport name in Khmer.

### Khmer Rendering

**#K01 · No Khmer-specific line-height override**
- File: `app/globals.css`
- Khmer script uses multi-level stacking vowels that require at least `line-height: 1.8` to avoid clipping. Tailwind default (`leading-normal`, 1.5) is insufficient. The `globals.css` has no `@layer base` rule targeting Khmer text with `line-height`. In tables with Khmer `name_kh` values, bottom-of-character strokes (e.g., ្ and subscript consonants) may be visually clipped.
- Recommended: add `.font-khmer, [lang="km"]` { `line-height: 1.8`; } to `globals.css`.

**#K02 · Battambang `display: swap` causes FOUT on first load**
- File: `app/layout.tsx:11–16`
- `display: 'swap'` means the system fallback font renders until Battambang downloads. For a Khmer-primary system, the fallback is likely a system serif (or even a Latin font that can't render Khmer at all), causing brief tofu boxes before the font loads.
- Recommended: `display: 'optional'` (no font shift) or pre-connect hint.

**#K03 · Sidebar app name `text-sm` may truncate long Khmer text**
- File: `modules/common/components/Sidebar.tsx:70`
- `nav.appName` = "ព្រឹត្តិការណ៍កីឡាជាតិ" (17 chars). At `text-sm` in a 224px sidebar, this may wrap onto 2 lines (especially with Khmer inter-character spacing). No `truncate` or `leading` guard.

**#K04 · Survey count `<label>` elements are `text-xs` — very small for Khmer**
- File: `modules/survey/components/BySportSurveyForm.tsx:84`
- Count field labels use `text-xs text-muted-foreground`. Khmer text at 12px is very hard to read (Unicode complex script at small sizes renders poorly on low-DPI screens). Recommend `text-sm` minimum for Khmer labels.

**#A06 · CardsPage filter `<select>` elements have no accessible label**
- File: `modules/cards/components/CardsPage.tsx:43–65`

**#A07 · Modal close button — verify `aria-label`**

## 5. Survey Data Integrity Bugs (Critical)

These are not UX style issues — they produce wrong data in the database.

### #C01 · BySportSurveyForm sends wrong `sports_id`

**File:** `modules/survey/components/BySportSurveyForm.tsx:48–55`

```tsx
submitMutation.mutate({
  org_id:          sport.id,           // ← sports_event association ID, not org ID
  events_id:       eventId,
  sports_id:       sport.id,           // ← same wrong ID used for both fields
  organization_id: organizationId,
  ...counts,
});
```

`SportsEventPublic` has an `id` field that is the event-sport association row ID — not the `sports_id` or `org_id`. The correct fields would come from a lookup. The comment in the file acknowledges this gap. Every survey entry submitted via BySportSurveyForm has incorrect `sports_id`.

### #C02 · BySportSurveyForm existing-entry lookup always returns `undefined`

**File:** `modules/survey/components/BySportSurveyForm.tsx:136–138`

```tsx
const getExistingEntry = (sportName: string | null): SurveyEntry | undefined =>
  eventEntries.find((e) => e.org_name === sportName);
```

This compares `e.org_name` (e.g., "ក្រសួងអប់រំ") against `sportName` (e.g., "ជណ្ដើររត់"). The comparison will always be false. Survey entries are never pre-populated into the form — users re-enter data on every visit and inadvertently create duplicates.
