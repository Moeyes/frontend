## 3. Gap Analysis — 10 Critical Paths

### A. Authentication — Login / Logout / Token Refresh / Expired Token

**Tested**: ❌ No  
**Risk**: High — government system; authentication failure = data breach  

**Untested behaviours:**
- Login with valid credentials → cookies set with correct flags (HttpOnly, Secure, SameSite=Strict)
- Login with wrong password → shows Khmer error message, form NOT locked
- `user_id` cookie is present after login → `AuthContext` bootstraps user state
- `access_token` expires → `authMiddleware` triggers silent refresh → original request retried
- Refresh fails (token invalid) → redirected to `/login`
- Logout → all three cookies deleted → user state cleared → `/login` redirect

**Recommended test type**: Integration (Next.js route handler + MSW for backend)

**Starter outline**:
```
describe('POST /api/auth/login')
  ✓ valid credentials → sets access_token, refresh_token (HttpOnly), user_id cookies
  ✓ invalid credentials → 401 response, no cookies set
  ✓ missing username → 400/422 passthrough from backend

describe('POST /api/auth/logout')
  ✓ deletes all three cookies
  ✓ calls backend /api/auth/logout with refresh_token (SEC-H6 fix)

describe('POST /api/auth/refresh')
  ✓ valid refresh_token → new access_token + refresh_token set
  ✓ missing refresh_token → 401, no cookies set
  ✓ invalid refresh_token → 401, all cookies deleted

describe('authMiddleware (client.ts)')
  ✓ 401 response → tryRefresh() called → original request retried
  ✓ 401 + refresh fail → window.location = '/login'
  ✓ 403 response → window.location = '/unauthorized'
```

### D. Age Computation — `computeAgeAtEvent`

**Tested**: ❌ No  
**Risk**: Critical — wrong age → wrong document requirement → legal/compliance failure  
**Location**: `core/lib/format.ts:60–79`

**Untested edge cases:**
- Born 2000-01-15, event 2018-01-15 → age = 18 (birthday IS the event day → adult)
- Born 2000-01-15, event 2018-01-14 → age = 17 (day before birthday → still minor)
- Born 2000-01-15, event 2018-01-16 → age = 18 (day after birthday → adult)
- Null DOB → returns null
- Null event date → returns null
- Invalid date string → returns null
- Leap year: born 2000-02-29, event 2018-02-28 → age = 17 (has not had birthday yet)
- Leap year: born 2000-02-29, event 2018-03-01 → age = 18

**Recommended test type**: Unit (pure function, no dependencies)

### E. Document Type Rule — `isMinorAtEvent` + DocumentStep

**Tested**: ❌ No  
**Risk**: Critical — wrong document required = registration blocked for valid athlete / invalid athlete passes  
**Location**: `core/lib/format.ts:81–88`, `modules/registration-flow/components/DocumentStep.tsx`

**Untested behaviours:**
- `isMinorAtEvent(dob, eventDate)` where age = 17 → returns true → birth cert required
- `isMinorAtEvent(dob, eventDate)` where age = 18 → returns false → NID/passport required
- `isMinorAtEvent(dob, null)` → returns false (safe default, age null → not minor)
- DocumentStep with `isMinor=true` → shows only birth cert field, hides NID/passport
- DocumentStep with `isMinor=false` → shows NID + passport fields, hides birth cert
- DocumentStep with `isMinor=null` (no event start date) → shows warning banner, shows photo only
- Form submission without birth cert when minor → blocked by Zod `superRefine`
- Form submission without NID/passport when adult → blocked by Zod `superRefine`

**Recommended test type**: Unit (isMinorAtEvent) + Component (DocumentStep with React Testing Library)

**Tested**: ❌ No

**Backend gaps exist** (see SCENARIOS.md gap table), but frontend logic is testable:

ReviewActions button states:
- Already `APPROVED` → Approve button disabled
- Already `REJECTED` → Reject button disabled
- Already `FLAGGED` → Flag button disabled
- Pending mutation → all three buttons disabled

**Recommended test type**: Unit (status mappings) + Component (ReviewActions button states)

## 7. Test Infrastructure Setup Recommendations

### Recommended Stack

| Tool | Purpose | Priority |
|---|---|---|
| **Vitest** | Unit + integration test runner — ESM-native, fast, Vite-compatible | Install first |
| **@testing-library/react** | Component tests — render, interact, assert | Install with Vitest |
| **@testing-library/user-event** | Realistic user interactions (type, click, tab) | Install with RTL |
| **MSW (Mock Service Worker)** | Mock backend API responses for unit/integration tests | Install second |
| **Playwright** | E2E browser automation | Install third |
| **jsdom / happy-dom** | DOM environment for Vitest component tests | Comes with Vitest |

**Why Vitest over Jest**: The project uses Next.js 16 + React 19 with ESM modules. Jest requires `transform` configuration to handle ESM. Vitest works natively with Vite's module resolution and is 2–5× faster on cold starts.

### Setup Steps

**Step 1 — Install Vitest + Testing Library**:
```bash
pnpm add -D vitest @vitest/coverage-v8 @testing-library/react @testing-library/user-event @testing-library/jest-dom happy-dom
```

**Step 2 — Add vitest.config.ts**:
```ts
// vitest.config.ts
import { defineConfig } from 'vitest/config';
import tsconfigPaths from 'vite-tsconfig-paths';

export default defineConfig({
  plugins: [tsconfigPaths()],
  test: {
    environment: 'happy-dom',
    globals: true,
    setupFiles: ['./vitest.setup.ts'],
    coverage: {
      provider: 'v8',
      thresholds: {
        lines: 70,
        branches: 60,
        functions: 70,
      },
      include: ['core/lib/**', 'modules/*/services/schema.ts'],
      exclude: ['**/*.d.ts', 'node_modules/**', '.next/**'],
    },
  },
});
```

**Step 3 — Add vitest.setup.ts**:
```ts
// vitest.setup.ts
import '@testing-library/jest-dom';
```

**Step 4 — Add scripts to package.json**:
```json
"scripts": {
  "test": "vitest",
  "test:coverage": "vitest run --coverage",
  "test:ui": "vitest --ui",
  "test:e2e": "playwright test"
}
```

**Step 5 — Install MSW for API mocking**:
```bash
pnpm add -D msw
```

**Step 6 — Install Playwright**:
```bash
pnpm add -D @playwright/test
pnpm exec playwright install chromium
```
