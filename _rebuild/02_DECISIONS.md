# Decisions log

Quick reference for "why did we do it this way."

## Scope
- Frontend-only rebuild. Backend (`Backend-V2/`) is teammate's work.
- Full rebuild from scratch in `final/`.
- Preserve only `messages/*.json`, `package.json`, standard Next.js config.

## Stack
- Next.js App Router + TypeScript
- shadcn/ui (re-init fresh)
- Tailwind CSS
- React Query for server state
- React Hook Form + Zod for forms
- next-intl for i18n
- TanStack Table for data tables
- pnpm only

## API layer
- Backend types auto-generated from OpenAPI via `openapi-typescript`
- Single typed HTTP client in `core/api/client.ts`
- All mutations send `Idempotency-Key` header (UUID v4)
- 401 → token refresh; 403 → redirect to `/unauthorized`
- RFC7807 error format expected from backend

## Auth
- Tokens in HttpOnly cookies via Next.js route handlers (NOT localStorage)
- Refresh-token rotation on every refresh
- AuthContext exposes user, roles, federation_id, organization_id

## RBAC
- 5 roles: Super Admin, Admin, Federation, Organization, Organizer
- `<ProtectedRoute requiredRoles={[...]}/>` on protected pages
- `useRequireRole` hook for component-level gating
- Federation/Org users see only own scoped data

## i18n
- Khmer is canonical. English secondary.
- All strings via `t('key')`. Zero hardcoded strings.
- Every key in BOTH `messages/en.json` AND `messages/kh.json`
- Battambang font for Khmer; Inter for English.

## State management
- Server state: React Query
- Form state: React Hook Form
- UI state: useState locally
- Cross-component UI state: Context (sparingly)
- No Redux, Zustand, Jotai

## Folder shape per module
```
modules/<domain>/
  components/    PascalCase .tsx + index.ts barrel
  hooks/         use<X>.ts files + index.ts
  services/      api calls typed via _contract/api.types.ts
  types/         domain-local types
  index.ts       public surface
```

## Data rules
- Age computed from event date, not today
- Document type: birth cert if <18, NID/passport if ≥18
- Per-sport quotas enforced at registration
- FSM transitions only via mutation endpoints (never client-side status changes)
- Audit logging on backend (frontend assumes this)

## Reports
- 8 Khmer reports generated by backend, frontend lists/triggers/downloads
- Backend returns signed URL with TTL; frontend follows redirect
- Async generation via job queue; frontend polls for status

## Out of scope (8-week plan)
- Mobile app
- Public-facing athlete/spectator pages
- Advanced analytics beyond role landing pages
- (Card features land in W8 if time permits)

## Things we explicitly chose NOT to do
- Not switching to Django
- Not splitting into microservices
- Not using Refine.dev or admin frameworks
- Not using Supabase (data sovereignty)
- Not using server actions for the API (REST only)

## 8-week strategy
- Two intermediate releases (W4 beta + W6 pilot) before W8 production
- Real federations test at each release
- Feedback time built into W5 and W7

## Vibe coding posture
- User does evening review only
- One module per evening, max
- Self-audit via SECURITY_CHECKLIST.md required before commit
- Hard rules in RED_LINES.md must never be violated
- Weekly review every Sunday via WEEKLY_REVIEW.md prompt
