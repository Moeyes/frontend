# Internationalization (i18n)

## Why Khmer-first

This is a ministry system for Cambodia. All users are Cambodian government staff, sport federation officials, and organization representatives who speak Khmer as their primary language. English is provided as a secondary option for technical staff.

**Rule:** All new strings go into `messages/kh.json` first. The English translation is required simultaneously — the i18n parity test (`core/lib/__tests__/i18n-parity.test.ts`) fails CI if the key counts differ.

---

## How next-intl is configured

**`i18n/request.ts`** — server-side locale detection:
```ts
// Reads locale from 'locale' cookie (set by LanguageProvider)
// Falls back to 'kh' (Khmer) if no cookie
```

**`app/layout.tsx`** — provides messages to React tree:
```tsx
<NextIntlClientProvider locale={locale} messages={messages}>
  {children}
</NextIntlClientProvider>
```

**In components:**
```ts
const t  = useTranslations('events');     // namespace
const tc = useTranslations('common');     // common namespace

t('title')              // → "ព្រឹត្តិការណ៍"
t('status.published')   // → "បានបោះផ្សាយ"
tc('save')              // → "រក្សាទុក"
```

## How to add a new key

1. Add to **both** files at the same time:

```json
// messages/en.json  (add under the appropriate namespace)
"mymodule": {
  "title":       "My Module",
  "description": "Manage my things"
}

// messages/kh.json
"mymodule": {
  "title":       "ម៉ូឌុលរបស់ខ្ញុំ",
  "description": "គ្រប់គ្រងរបស់ខ្ញុំ"
}
```

2. Use in the component:
```ts
const t = useTranslations('mymodule');
<PageHeader title={t('title')} description={t('description')} />
```

3. Run the parity test:
```bash
pnpm test:run  # fails if en.json and kh.json key counts differ
```

### Keys with variables

```json
"welcomeBack": "Welcome back, {name}"     // en
"welcomeBack": "សូមស្វាគមន៍មកវិញ {name}" // kh
```

```ts
t('welcomeBack', { name: user.kh_given_name })
```

### Nested namespaces

Messages are organized by module namespace. Always add new keys under the relevant namespace:

```
common.*        — buttons, labels, errors used everywhere
nav.*           — sidebar navigation labels
auth.*          — login page
dashboard.*     — dashboard page
events.*        — event management
sports.*        — sports management
organizations.* — organization management
users.*         — user management
registration.*  — participant registration
participation.* — organizer registration
survey.*        — survey forms
submissions.*   — admin review queue
reports.*       — report generation
cards.*         — participant cards
```

## Khmer fonts

**Battambang** font is loaded from Google Fonts in `app/layout.tsx`:

```tsx
import { Battambang } from 'next/font/google';

const battambang = Battambang({
  subsets: ['khmer'],
  weight: ['400', '700'],
  variable: '--font-battambang',
});
```

The font is applied to the entire document via the CSS variable. Tailwind uses `font-sans` which resolves to Battambang for Khmer characters and Inter for Latin.

**Important:** Native `<select>` and `<option>` elements on Windows may not use Battambang — the OS renders them in its system font. Use the shadcn `Select` component for dropdowns that contain Khmer text.

## Khmer numerals

Use `toKhmerNumerals()` from `core/lib/format` when displaying numbers in a Khmer context:

```ts
import { toKhmerNumerals } from '@/core/lib';

toKhmerNumerals(2024)   // → '២០២៤'
toKhmerNumerals('42')   // → '៤២'
```

For numbers in data tables (counts, IDs), the standard is to leave them as Arabic numerals with the `tabular-nums` CSS class for alignment. For year displays or prominent counts, Khmer numerals are preferred.

## Date and time formatting

Always use `formatDate()` and `formatDateTime()` from `core/lib/format`:

```ts
import { formatDate, formatDateTime } from '@/core/lib';

// kh locale (default)
formatDate('2024-03-15')          // → Khmer locale format
formatDate('2024-03-15', 'en')    // → '15/03/2024' (dd/MM/yyyy)

formatDateTime('2024-03-15T09:30:00', 'en')  // → '15/03/2024, 09:30'
```

- Returns `'—'` for null, undefined, or invalid dates
- Uses `Intl.DateTimeFormat` — respects locale

**Age computation:** Never use `new Date()` for age. Always use `computeAgeAtEvent(dob, event.start_date)`. This is Red Line #3 — see `_rebuild/RED_LINES.md`.

## Language switcher

The `LanguageSwitcher` component in the top bar switches the `locale` cookie and triggers a full page reload. The cookie is read by `i18n/request.ts` on the next server render.

To switch programmatically:
```ts
import { useLanguage } from '@/core/i18n';

const { locale, setLocale } = useLanguage();
setLocale('en');  // or 'kh'
```

## PDF and Excel Khmer rendering notes

- **Excel reports** are generated by the backend. Khmer in Excel cells requires the backend to write UTF-8 encoded cell data with a compatible font set. The frontend downloads the binary blob as-is.
- If Khmer text appears as `???` or boxes in Excel on Windows, the root cause is usually the backend not setting the correct font on cells. This is a backend concern — file an issue tagged `backend`.
- **PDF generation** is not currently implemented in the frontend. If added, use a PDF library that supports Unicode fonts, not canvas-based rendering.
